## 2. 全局设计规范

### 2.1 命名规范

| 对象 | 命名规范 | 示例 |
|:---|:---|:---|
| **数据库** | 小写英文，项目名缩写 | `zhuanzhuan` |
| **表名** | 小写英文单数，下划线分隔 | `user`, `order`, `product_image` |
| **字段名** | 小写英文，下划线分隔 | `user_id`, `created_at` |
| **主键** | `PK_表名` | `PK_user` |
| **唯一键** | `uk_字段名` | `uk_email` |
| **索引** | `idx_字段名` | `idx_user_id` |
| **外键** | `fk_子表_父表` | `fk_order_user_buyer` |

### 2.2 公共字段规范

所有业务表均包含以下公共字段：

| 字段名 | 类型 | 约束 | 说明 |
|:---|:---|:---:|:---|
| `id` | BIGINT(20) | PK, AUTO_INCREMENT | 主键ID |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 更新时间 |
| `deleted_at` | DATETIME | DEFAULT NULL | 删除时间（软删除标记） |

> **注意：** `deleted_at` 为 NULL 表示未删除，非 NULL 表示已删除。所有业务查询均需附加 `deleted_at IS NULL` 条件。

### 2.3 字段类型规范

| 数据类型 | 使用场景 |
|:---|:---|
| `BIGINT(20)` | 主键ID、外键ID |
| `VARCHAR(50)` | 用户名、分类名、收件人 |
| `VARCHAR(100)` | 邮箱、标题 |
| `VARCHAR(255)` | URL、密码哈希 |
| `VARCHAR(500)` | 较长的文本内容 |
| `TEXT` | 富文本描述、长文本 |
| `DECIMAL(10,2)` | 金额（精确到分） |
| `INT(11)` | 计数（浏览量、收藏数） |
| `TINYINT(4)` | 状态标记（0/1） |
| `ENUM` | 枚举值（状态、类型） |
| `DATETIME` | 时间戳 |

### 2.4 存储引擎与字符集

```sql
-- 建库语句
CREATE DATABASE IF NOT EXISTS zhuanzhuan
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

-- 建表均使用 InnoDB 引擎 + utf8mb4 字符集
ENGINE = InnoDB
DEFAULT CHARSET = utf8mb4
COLLATE = utf8mb4_unicode_ci;
```

## 4. 数据库表详细设计

### 4.1 用户表 (user)

存储系统用户的注册信息与账户状态。

| 字段名 | 类型 | 长度 | 允许空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---:|:---|
| id | BIGINT | 20 | NO | AUTO_INCREMENT | 用户ID（主键） |
| username | VARCHAR | 50 | NO | | 用户名（唯一） |
| password_hash | VARCHAR | 255 | NO | | 密码哈希值（BCrypt） |
| email | VARCHAR | 100 | YES | NULL | 邮箱（唯一） |
| phone | VARCHAR | 20 | YES | NULL | 手机号（唯一） |
| avatar | VARCHAR | 255 | YES | NULL | 头像URL |
| nickname | VARCHAR | 50 | YES | NULL | 昵称 |
| bio | VARCHAR | 200 | YES | NULL | 个人简介 |
| gender | TINYINT | 4 | YES | 0 | 性别（0未知/1男/2女） |
| role | ENUM | — | NO | 'user' | 角色（user/admin） |
| credit_score | INT | 11 | YES | 100 | 信誉分 |
| status | TINYINT | 4 | NO | 1 | 状态（0禁用/1启用） |
| last_login_at | DATETIME | — | YES | NULL | 最后登录时间 |
| created_at | DATETIME | — | NO | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | — | NO | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |
| deleted_at | DATETIME | — | YES | NULL | 删除时间（软删除） |

**索引设计：**

| 索引名 | 类型 | 字段 | 说明 |
|:---|:---:|:---|:---|
| `PRIMARY` | PRIMARY | id | 主键索引 |
| `uk_username` | UNIQUE | username | 用户名唯一 |
| `uk_email` | UNIQUE | email | 邮箱唯一 |
| `uk_phone` | UNIQUE | phone | 手机号唯一 |
| `idx_status` | INDEX | status | 状态查询加速 |

**初始化数据：**

```sql
-- 默认管理员
INSERT INTO user (username, password_hash, email, role, status)
VALUES ('admin', '$2a$10$...', 'admin@zhuanzhuan.com', 'admin', 1);
```

## 6. Redis 缓存设计

### 6.1 缓存策略总表

| 缓存对象 | Key 模式 | 数据类型 | 过期时间 | 更新策略 |
|:---|:---|:---|:---:|:---|
| **验证码** | `captcha:{email/phone}` | String | 5分钟 | 生成时写入 |
| **用户会话** | `session:{userId}` | String (JSON) | 7天 | 登录时写入 |
| **热点商品详情** | `product:{id}` | String (JSON) | 30分钟 | Cache-Aside |
| **商品分类树** | `category:tree` | String (JSON) | 24小时 | Cache-Aside |
| **首页推荐列表** | `index:recommend` | ZSet | 10分钟 | 定时刷新 |
| **商品浏览量** | `view:product:{id}` | String | 持久化 | 定时刷入DB |
| **接口频率限制** | `rate:limit:{ip}:{endpoint}` | String | 1分钟 | 请求时自增 |

### 6.2 缓存更新模式

采用 **Cache-Aside Pattern**：

```
读取路径：
  read cache → hit? → return cached data
             → miss? → read database → write cache → return

写入路径：
  write database → delete cache → 完成
```

## 7. SQL 优化建议

### 7.1 索引优化

- **高频查询加索引：** 对 `product.status`、`order.status`、`product.created_at` 等查询条件字段建立索引
- **联合索引：** 对 `(status, created_at)` 等组合查询条件建立联合索引
- **覆盖索引：** 对频繁查询的字段组合建立覆盖索引，避免回表查询
- **全文索引：** 商品表的 `title` 和 `description` 字段建立全文索引，用于搜索
- **避免冗余索引：** 定期检查并删除重复或未被使用的索引

### 7.2 查询优化

- 避免 `SELECT *`，只查询需要的字段
- 使用 `EXPLAIN` 分析 SQL 执行计划，确保走索引
- 分页查询使用覆盖索引 + 子查询方式优化
- 大数据量统计使用缓存或异步计算

### 7.3 数据库维护

- 定期执行 `OPTIMIZE TABLE` 回收碎片空间
- 监控慢查询日志，优化慢 SQL
- 每日自动备份，保留最近 7 天备份
- 对大表按月或按季度进行分区
